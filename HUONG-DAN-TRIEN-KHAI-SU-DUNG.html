1Tổng quan hệ thống

Hệ thống cho phép một người dùng đẩy một luồng video duy nhất (từ vMix, OBS, hoặc bất kỳ công cụ nào hỗ trợ RTMP/SRT) lên máy chủ, sau đó tự động phân phối (fan-out) ra nhiều nền tảng cùng lúc — Facebook, YouTube, hoặc đích RTMP tùy chỉnh — kèm theo ghi hình, xem trước (preview) và nguồn dự phòng.

Mỗi buổi phát được mô hình hóa thành một “sự kiện” (event). Người tổ chức tạo sự kiện trong trang quản trị rồi giao cho người dùng 3 thứ:

  • Khóa ingest bí mật để cấu hình vMix/OBS.
  • Link cấu hình (kèm mã đăng nhập) để người dùng tự thêm/sửa các đích phát.
  • Link trạng thái chỉ-đọc để chia sẻ cho người theo dõi buổi phát.
Nhiều sự kiện chạy song song Hệ thống hỗ trợ nhiều người dùng tổ chức nhiều buổi phát đồng thời — mỗi sự kiện độc lập, có khóa và link riêng.

2Kiến trúc & khái niệm cốt lõi

Khái niệmÝ nghĩa
Sự kiện = 1 path MediaMTX Tên path là một chuỗi bí mật ~20 ký tự — chính là khóa ingest. Không thể đoán nên không cần mật khẩu publish riêng.
Lưu metadata trong path config Toàn bộ thông tin sự kiện (tên, token, danh sách đích, mã đăng nhập đã băm…) được nhúng trong trường runOnReady của path dưới dạng comment # RELAY_META:<base64>. Không dùng cơ sở dữ liệu riêng.
Fan-out Khi có nguồn, MediaMTX chạy ffmpeg … -f tee để sao chép luồng ra tất cả đích đang bật (không re-encode, -c copy).
Trang public ẩn danh Hai trang ngoài khu vực đăng nhập admin: /public/config/<token> và /public/status/<token>. Mọi truy vấn đi qua API server-side, dùng thông tin admin phía máy chủ — không lộ thông tin đăng nhập MediaMTX cho khách.
Token phân quyền configToken (cần mã đăng nhập, cho phép chỉnh sửa) và statusToken (chỉ đọc). Mỗi token có thể xoay (rotate) khi cần thu hồi.
Quan trọng về codec Vì fan-out dùng -c copy (không mã hóa lại để tiết kiệm CPU), nguồn đẩy vào phải là H.264 + AAC đúng chuẩn mọi nền tảng yêu cầu. Hãy cấu hình vMix/OBS xuất H.264 (video) + AAC (audio).

3Yêu cầu hạ tầng & cổng mạng

Backend chạy bằng Docker. Cần mở các cổng sau trên máy chủ MediaMTX:

CổngGiao thứcVai trò
1935TCPRTMP — ingest từ vMix/OBS và đẩy ra nền tảng
8554TCPRTSP — nguồn nội bộ cho fan-out
8890UDPSRT — ingest độ trễ thấp (tùy chọn)
8888TCPHLS — xem trước (preview) qua proxy
8889 + 8189TCP + UDPWebRTC (tùy chọn)
9997TCPControl API — frontend gọi vào (có Basic Auth)
9998TCPMetrics (nếu bật)
Image MediaMTX BẮT BUỘC phải có ffmpeg Fan-out hoạt động bằng cách MediaMTX tự chạy ffmpeg bên trong container server. Image mặc định bluenviron/mediamtx:latest là distroless — KHÔNG có ffmpeg và KHÔNG có shell, khiến fan-out “im lặng không làm gì”. Phải dùng bluenviron/mediamtx:latest-ffmpeg (nền Alpine, có sẵn ffmpeg 8.x và /bin/sh).

4Triển khai backend (MediaMTX)

File docker-compose gồm hai service: máy chủ MediaMTX và publisher (luồng test, tùy chọn).

4.1. File docker-compose mẫu

services:
  mediamtx:
    # BẮT BUỘC dùng tag -ffmpeg để fan-out chạy được
    image: bluenviron/mediamtx:latest-ffmpeg
    container_name: mediamtx
    restart: unless-stopped
    ports:
      - "8554:8554"        # RTSP
      - "1935:1935"        # RTMP
      - "8888:8888"        # HLS
      - "8889:8889"        # WebRTC (TCP)
      - "8189:8189/udp"    # WebRTC (UDP)
      - "8890:8890/udp"    # SRT (UDP)
      - "9997:9997"        # Control API
      - "9998:9998"        # Metrics
    volumes:
      - ./mediamtx.yml:/mediamtx.yml:ro
      - ./recordings:/recordings
      - ./data:/data
    environment:
      - TZ=UTC

  # (Tùy chọn) Luồng test để kiểm thử — đẩy testsrc vào path "test"
  publisher:
    image: bluenviron/mediamtx:latest-ffmpeg
    container_name: mediamtx-publisher
    restart: unless-stopped
    depends_on:
      - mediamtx
    # Entrypoint của image là binary mediamtx — phải override sang ffmpeg
    entrypoint: ffmpeg
    command: >
      -re -f lavfi -i testsrc2=size=1280x720:rate=30
      -f lavfi -i sine=frequency=440
      -c:v libx264 -preset ultrafast -tune zerolatency -g 30 -b:v 1000k
      -c:a aac -b:a 128k -f rtsp
      rtsp://mediamtx:8554/test
Hai lưu ý dễ sai ở service publisher
  • Phải có entrypoint: ffmpeg. Nếu thiếu, các tham số -re … sẽ bị truyền cho binary mediamtx → lỗi unknown flag -r.
  • Dùng sine=frequency=440 (không kèm duration=1), nếu không âm thanh sẽ tắt sau 1 giây.

4.2. Xác thực Control API

File mediamtx.yml nên bật authMethod: internal với tối thiểu:

  • Người dùng admin (mật khẩu riêng) có quyền api — frontend dùng để quản trị path.
  • Người dùng any (ẩn danh) có quyền publish + read + playback — để vMix/OBS đẩy và HLS đọc không cần mật khẩu (khóa ingest đã bí mật).

4.3. Khởi chạy & kiểm tra

# Trên máy chủ backend
docker compose pull
docker compose up -d

# Xác nhận server có ffmpeg (PHẢI in ra version)
docker compose exec mediamtx ffmpeg -version

# Kiểm tra Control API
curl -u admin:<mật_khẩu> http://<IP>:9997/v3/paths/list

5Cấu hình & chạy frontend (MTX-UI)

Tạo file .env.local (môi trường dev) hoặc .env (production/Docker) trỏ về backend:

Biến môi trườngÝ nghĩa
MEDIAMTX_API_URLURL Control API mà máy chủ frontend gọi vào, ví dụ http://<IP>:9997
MEDIAMTX_ADMIN_USER / MEDIAMTX_ADMIN_PASSThông tin admin để frontend ghi cấu hình path (chỉ dùng phía server, không lộ ra trình duyệt)
MEDIAMTX_HLS_URL / NEXT_PUBLIC_MEDIAMTX_HLS_URLURL máy chủ HLS, ví dụ http://<IP>:8888 (dùng cho preview qua proxy)
NEXT_PUBLIC_MEDIAMTX_RTMP_HOST / _RTMP_ADDRESSHostname & địa chỉ RTMP public để hiển thị URL ingest cho người dùng, ví dụ <IP> và <IP>:1935
NEXT_PUBLIC_MEDIAMTX_SRT_HOST / _SRT_ADDRESSTương tự cho SRT, ví dụ <IP> và <IP>:8890
RELAY_SESSION_SECRETChuỗi bí mật ngẫu nhiên để ký cookie phiên của trang cấu hình public. Bắt buộc đặt ở production.
Không commit thông tin nhạy cảm File .env.local chứa mật khẩu admin & secret — không được đưa lên git. Ở production, đặt RELAY_SESSION_SECRET bằng chuỗi ngẫu nhiên đủ dài và bật HTTPS (cookie phiên sẽ có cờ Secure).

Chạy frontend

# Phát triển
npm install
npm run dev          # mặc định http://localhost:3000

# Production (Docker)
docker compose up -d dashboard

6Hướng dẫn cho Admin: tạo sự kiện

Trong dashboard, mở tab “Paths” → thẻ “Sự kiện relay (public)”.

1
Nhập Tên sự kiện (ví dụ tên người tổ chức) và Quota luồng (số đích tối đa được bật cùng lúc, mặc định 10).
2
Bấm “Tạo sự kiện”. Hệ thống sinh ra một path bí mật + các token.
3
Màn hình hiện 4 ô (đều có nút Copy):
  • Khóa ingest (path) — đưa cho người dùng cấu hình vMix/OBS.
  • Mã đăng nhập — để mở trang cấu hình. Chỉ hiển thị MỘT LẦN
  • Link cấu hình — /public/config/<configToken>
  • Link trạng thái — /public/status/<statusToken>
Mã đăng nhập không thể khôi phục Hệ thống chỉ lưu bản băm (salted hash). Nếu làm mất mã, người dùng phải dùng chức năng “Tạo lại mã đăng nhập” trong trang cấu hình (cần đang có phiên) hoặc admin tạo sự kiện mới.

7Hướng dẫn người dùng: cấu hình & phát

Bước 1 — Cấu hình vMix / OBS

Dùng Khóa ingest admin đã cấp. Trỏ phần mềm phát đến:

RTMP:

URL:   rtmp://<IP>:1935
Khóa:  <khóa-ingest>

Hoặc SRT (độ trễ thấp): dùng địa chỉ srt://<IP>:8890 với streamid=publish:<khóa-ingest>.

Cấu hình mã hóa: Video H.264, Audio AAC.

Bước 2 — Mở trang cấu hình

Truy cập Link cấu hình, nhập mã đăng nhập. Phiên làm việc được giữ bằng cookie an toàn (HttpOnly).

Tại đây người dùng tự quản lý toàn bộ buổi phát mà không cần đụng đến trang admin.

Các chức năng trong trang cấu hình

Chức năngMô tả
Thêm / sửa / xóa đích phátMỗi đích gồm: tên, nền tảng (Facebook/YouTube/Custom), server URL, stream key, bật/tắt. Số đích đang bật bị giới hạn bởi quota.
Bật / tắt relayTạm dừng toàn bộ fan-out mà vẫn giữ cấu hình các đích.
Bật / tắt ghi hìnhGhi lại buổi phát trên máy chủ.
Xoay khóa ingest (rotate stream id)Đổi khóa ingest mới (thu hồi khóa cũ); link cấu hình/trạng thái giữ nguyên.
Đổi mã đăng nhập / tạo lại mãThay đổi hoặc cấp lại mã vào trang cấu hình.
Xoay token trạng thái / cấu hìnhThu hồi link đã chia sẻ bằng cách cấp token mới.
Nguồn dự phòng (fallback)Cấu hình nội dung thay thế (chữ/ảnh/video) khi mất nguồn chính.
Xem trước (preview)Xem luồng HLS trực tiếp qua proxy có kiểm soát bằng token (không lộ host nội bộ).
Quy trình điển hình của một buổi phát
  1. Người dùng nhận khóa ingest + 2 link từ admin.
  2. Mở trang cấu hình, thêm các đích (FB/YT/…) và bật relay.
  3. Cấu hình vMix/OBS bằng khóa ingest, bắt đầu phát.
  4. Trạng thái chuyển Online, fan-out tự đẩy ra mọi đích đang bật; theo dõi qua preview.
  5. Chia sẻ link trạng thái cho người liên quan.

8Trang trạng thái (chia sẻ)

Link trạng thái (/public/status/<statusToken>) là trang chỉ đọc, không cần đăng nhập, dùng để chia sẻ rộng. Nó hiển thị:

  • Tên sự kiện và trạng thái nguồn (Online/Offline).
  • Bản xem trước (preview) khi đang phát.
  • Danh sách đích phát với stream key đã được che (masked).
  • Thông số luồng (số track, lưu lượng) cập nhật theo chu kỳ.
Bảo mật trang trạng thái Trang trạng thái không bao giờ lộ khóa ingest, URL đẩy đầy đủ, hay stream key thật. Người xem chỉ biết buổi phát đang chạy, không thể chiếm quyền hay chen luồng.

9Xử lý sự cố thường gặp

Triệu chứngNguyên nhân & cách khắc phục
Nguồn Online nhưng các nền tảng không nhận được luồng (fan-out “im lặng”) Image server không có ffmpeg. Kiểm tra docker compose exec mediamtx ffmpeg -version. Nếu lỗi → đổi image sang bluenviron/mediamtx:latest-ffmpeg, rồi docker compose pull mediamtx && docker compose up -d --force-recreate mediamtx.
Container publisher báo mediamtx: error: unknown flag -r Thiếu entrypoint: ffmpeg ở service publisher → tham số ffmpeg bị truyền cho binary mediamtx. Thêm dòng entrypoint: ffmpeg.
Âm thanh luồng test tắt sau ~1 giây Bộ lọc sine=…:duration=1 chỉ phát 1 giây. Bỏ duration=1.
Preview HLS lỗi 401 / không phát được Proxy phải chuyển tiếp query-string ?session=… của LL-HLS. Bản hiện tại đã xử lý; nếu tự sửa, đảm bảo route HLS giữ nguyên search khi gọi upstream.
SRT không kết nối Cổng 8890 phải map dạng UDP (8890:8890/udp), không phải TCP.
Truy cập link sai → 404 Đúng thiết kế. Token không khớp hoặc nhầm loại (configToken trên endpoint status) đều trả 404 để tránh dò quét.
Ghi chú kỹ thuật (cho người bảo trì) MediaMTX chạy runOnReady bằng exec trực tiếp có tách tham số theo dấu nháy, KHÔNG qua shell. Do đó # không phải comment và : không phải lệnh rỗng. Hệ thống bọc lệnh trong sh -c '…' (nhờ image -ffmpeg có /bin/sh) để comment metadata và no-op hoạt động đúng. Với ffmpeg 8.x, timeout đọc RTSP dùng cờ -timeout (không phải -rw_timeout).
